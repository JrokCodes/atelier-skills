# Screenshot Loop

Take screenshots of the build, compare against the design brief or reference, identify differences, fix them, and repeat. This is the core quality assurance loop for every frontend build.

---

## What Is the Screenshot Loop?

The screenshot loop is an iterative QA process:

1. **Screenshot** the current build at multiple breakpoints
2. **Compare** against the design brief, reference site, or previous iteration
3. **Identify** specific differences (measurements, colors, spacing, typography)
4. **Fix** the issues in code
5. **Re-screenshot** and compare again
6. **Repeat** until the build matches the design intent

This loop runs after every major build iteration, before client review, and before deploy. Claude is multimodal and can directly read screenshot images, making this a natural part of the development workflow.

## Screenshot Tool

**Location:** `Atelier/tools/screenshot/screenshot.js`

**First-time setup:**
```bash
cd Atelier/tools/screenshot && npm install
```

### Basic Usage

```bash
node Atelier/tools/screenshot/screenshot.js <url> <output.png> [flags]
```

### Flags

| Flag | Purpose | Example |
|------|---------|---------|
| `--full-page` | Capture the entire scrollable page | `--full-page` |
| `--width=N` | Set viewport width in pixels | `--width=375` |
| `--delay=N` | Wait N milliseconds before capture | `--delay=2000` |

### Standard Breakpoint Screenshots

Always capture at these three breakpoints:

```bash
# Mobile (375px)
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/mobile.png --full-page --width=375

# Tablet (768px)
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/tablet.png --full-page --width=768

# Desktop (1440px)
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/desktop.png --full-page --width=1440
```

### Important Limitations

**DO NOT screenshot animated elements.** Animations freeze mid-frame in screenshots, producing misleading results. Specifically:
- Entrance animations captured mid-fade look broken
- Scroll-triggered animations won't trigger in a static capture
- Loading spinners and skeleton states will appear instead of content
- Hover states can't be captured reliably

**Use `--delay=2000` for pages with entrance animations.** This waits 2 seconds for animations to complete before capturing:

```bash
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/desktop.png --width=1440 --delay=2000
```

If the page has staggered animations (elements appearing one after another), increase the delay:
```bash
# 4 second delay for pages with staggered entrance animations
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/desktop.png --width=1440 --delay=4000
```

## Comparison Workflow

### Step 1: Take Screenshots at All 3 Breakpoints

```bash
# Create screenshots directory
mkdir -p screenshots

# Capture all three
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/mobile.png --full-page --width=375 --delay=2000
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/tablet.png --full-page --width=768 --delay=2000
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/desktop.png --full-page --width=1440 --delay=2000
```

### Step 2: Read the Screenshot Images

Claude is multimodal. Read the screenshots directly:

```
Read screenshots/desktop.png
Read screenshots/mobile.png
Read screenshots/tablet.png
```

This renders the images visually, allowing direct comparison.

### Step 3: Compare Against References

Compare against one or more of:
- **Design brief** (DESIGN-BRIEF.md generated by the design-advisor skill)
- **Reference site screenshot** (captured from the reference URL)
- **Previous iteration** (to verify fixes worked)
- **Figma mockup** (if the client provided one)

To screenshot a reference site:
```bash
node Atelier/tools/screenshot/screenshot.js https://reference-site.com screenshots/reference-desktop.png --full-page --width=1440 --delay=3000
```

### Step 4: Generate a Specific Diff List

Don't say "looks a bit off." Be precise:

**Good diff list:**
```
DESKTOP (1440px):
1. Hero heading is ~32px, brief specifies 56px — increase font size
2. CTA button has 12px padding, should be 16px 32px per brief
3. Section spacing is ~48px, design shows ~80px gap between sections
4. Card grid is 4 columns, brief shows 3 columns with larger cards
5. Footer background is #111111, should be #0A0A0A to match hero

MOBILE (375px):
1. Hero heading overflows — needs max-width or smaller font at this breakpoint
2. Card stack has no gap — add 16px gap between cards
3. CTA button is full-width but text is too small (14px → 16px)
```

**Bad diff list:**
```
- Doesn't look right
- Needs more spacing
- Colors seem off
```

### Step 5: Fix Issues and Re-Screenshot

After making fixes, re-capture and compare again:

```bash
# Re-capture after fixes
node Atelier/tools/screenshot/screenshot.js http://localhost:5173 screenshots/desktop-v2.png --full-page --width=1440 --delay=2000
```

Read both versions to confirm fixes:
```
Read screenshots/desktop.png      # Before
Read screenshots/desktop-v2.png   # After
```

### Step 6: Repeat Until Matching

Continue the loop until the build matches the design intent. Typically 2-4 iterations for a new page, 1-2 for fixes.

## Anti-AI-Slop Visual Check

Run this checklist after every build. AI-generated websites share telltale patterns that make them look generic. This checklist catches them.

### Typography Check
- [ ] **Display/heading fonts are NOT Inter, Roboto, or Arial.** These are the default AI choices. Use Satoshi, Cabinet Grotesk, Clash Display, General Sans, or other distinctive fonts. (Inter is fine for body text only.)
- [ ] **Font sizes create clear hierarchy.** Hero heading should be noticeably larger than section headings. If everything looks the same size, the hierarchy is flat.
- [ ] **Line heights are comfortable.** Body text: 1.5-1.7. Headings: 1.1-1.3. Not the default 1.0 or 2.0.
- [ ] **Letter spacing is intentional.** Headings may use slight negative tracking (-0.02em). Body text should be default (0). ALL-CAPS text needs positive tracking (+0.05em to +0.1em).

### Color Check
- [ ] **Colors are deliberate, not generic.** No default purple/blue gradient. Colors should come from the brand palette or design brief.
- [ ] **Maximum 2 accent colors.** Primary accent + one secondary. More than that looks chaotic.
- [ ] **Gradient usage is restrained.** Gradients on CTAs, accent borders, or overlays only. Never gradient backgrounds on full sections.
- [ ] **Dark mode uses semantic layering.** Background layers: #0A0A0A (base) → #111111 (surface) → #1A1A1A (elevated) → #222222 (overlay). Not flat #000000 everywhere.

### Layout Check
- [ ] **Layout is NOT perfectly symmetrical.** Perfect symmetry reads as AI-generated. Introduce intentional asymmetry: offset grids, varied card sizes, text aligned left while image is right.
- [ ] **There is a clear visual hero.** Something on the page is clearly the most important element. If everything is the same visual weight, nothing stands out.
- [ ] **Whitespace is generous.** AI tends to pack elements too tightly. Sections should breathe. 80-120px between major sections on desktop.
- [ ] **Content blocks have varied layouts.** Not every section is centered text + grid below. Mix: left-aligned text + right image, full-width feature, offset testimonial.

### Animation Check
- [ ] **Animations are subtle and purposeful.** Entrance animations: gentle fade + small translate (20px max). Duration: 0.4-0.6s. Easing: ease-out.
- [ ] **No bouncy/springy defaults.** If using Framer Motion spring, use `stiffness: 100, damping: 15` minimum. Not the default bouncy spring.
- [ ] **Stagger is fast.** If elements stagger in, 0.05-0.1s between each. Not 0.3s+ (feels like a slideshow).
- [ ] **Scroll animations trigger once.** Use `once: true` in Framer Motion's `whileInView`. Elements that re-animate every scroll feel gimmicky.

### Content Check
- [ ] **No placeholder text.** No "Lorem ipsum", no "Your Company Name Here", no "Description goes here."
- [ ] **CTA text is specific.** "Book a Free Consultation" not "Get Started" or "Learn More." The CTA should say what happens when you click.
- [ ] **Social proof is real.** Real names, real companies, real numbers. Or clearly marked as placeholder with a TODO.

## When to Run the Screenshot Loop

| Trigger | Breakpoints | Full Page? |
|---------|------------|------------|
| After building a new page | All 3 (375, 768, 1440) | Yes |
| After fixing layout issues | The affected breakpoint(s) | Relevant section |
| Before client review | All 3 | Yes |
| Before deploying to production | All 3 | Yes |
| After image optimization | Desktop (1440) | Hero section |
| After font changes | All 3 | Above the fold |

## Troubleshooting

### Screenshot is blank or white
- The dev server may not be running. Start it: `npm run dev`
- The URL may be wrong. Verify `http://localhost:5173` is accessible.
- Add `--delay=3000` to wait for initial render.

### Screenshot shows loading state
- The page has async data. Increase `--delay` to wait for data to load.
- If the page requires authentication, screenshot a public page or mock the data.

### Animations look frozen mid-frame
- Expected behavior. Increase `--delay` so animations complete before capture.
- For infinite animations (pulsing, rotating), they will always look frozen. Evaluate them in the browser instead.

### Colors look different in screenshot vs browser
- Screenshots use sRGB color space. Browser may display in P3.
- Compare hex values in code, not visual appearance in screenshots, for precise color matching.

### Mobile screenshot looks zoomed in
- Check that the page has a proper viewport meta tag:
  ```html
  <meta name="viewport" content="width=device-width, initial-scale=1" />
  ```
